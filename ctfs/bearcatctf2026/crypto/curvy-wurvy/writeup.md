## Solution

### Analysis

The challenge provides `curvy_wurvy.py`, an Ed25519 signing server with the following components:

**Key Derivation Function (KDF):**
```python
def KDF(uid):
    user_key = MASTER_KEY * uid % p
    return user_key
```
This is a **linear** KDF -- the user key is simply `MASTER_KEY * uid mod p` (where `p` is the Ed25519 group order).

**Sign Function (with a critical bug):**
```python
def sign(y, key):
    padding = 0
    while True:
        try:
            x = E.x_recover(y + padding)
            break
        except AssertionError:
            padding += 1
    P = Point(x, y, E)  # BUG: uses original y, not y+padding
    signed_point = E.mul_point(key, P)
    return signed_point.y + (padding << 256)
```
When `padding > 0`, the function recovers `x` from `y + padding` but constructs the point `P(x, y)` -- creating a point that is **not on the Ed25519 curve**. However, since ecpy's `_add_ext` and `_dbl_ext` functions only use the curve parameter `a` (not `d`), scalar multiplication still works consistently for off-curve points.

**Server Interface:**
1. **Get user id**: Returns a random UID in `[1, uid_bound]` where `uid_bound = int(pow(p, 1/4))` (approximately 2^63).
2. **Sign data**: Signs arbitrary data `y` with any UID using `KDF(uid)`.
3. **Verify signature**: Verifies a signature.

The server prints `FLAG_UID` and `flag_sig = sign(FLAG_as_int, KDF(FLAG_UID))` at startup.

### Approach

This was a long and complex solve with several pivots. The final successful approach used a **torsion oracle binary search**.

**Step 1: Identifying the Vulnerability**

The linear KDF means: if we recover MASTER_KEY (`M`), we can compute `flag_key = M * FLAG_UID % p`, invert it, and multiply the signed flag point by the inverse to recover the original FLAG point (and thus the flag bytes).

**Step 2: ecpy Internals Analysis**

A critical analysis of the ecpy library's extended coordinate arithmetic revealed:
- `_add_ext(X1,Y1,Z1,XY1, X2,Y2,Z2,XY2, q, a)` only uses `q` (field prime) and `a` (curve parameter), **not** `d`.
- `_dbl_ext(X1,Y1,Z1, q, a)` similarly only uses `q` and `a`.

This means the arithmetic is consistent even for points not on Ed25519, since `d` never appears in the computation. Points are effectively on a family of curves parameterized by `d`, but the arithmetic doesn't distinguish between them.

**Step 3: Torsion Point Oracle (Key Breakthrough)**

Ed25519's cofactor is 8, meaning there are torsion points of order 8. Let `T8` be a point of order 8. When we query `sign(T8_y, uid)`:

- The server computes `key = M * uid % p`
- Then computes `key * T8`
- Since `T8` has order 8, `key * T8 = (key mod 8) * T8`
- The result reveals `(M * uid mod p) mod 8`

This creates an oracle: for any `uid`, we learn `(M * uid mod p) mod 8`.

**Step 4: Initial Wrong Assumption and Pivot**

Initially, it was assumed that `MASTER_KEY` was bounded by `uid_bound` (approximately 2^63). A baby-step giant-step (BSGS) approach was attempted, even implementing a multi-threaded C version for speed. This search of the `[0, 2^63]` range found nothing.

After extensive debugging and analysis, the crucial realization was that `uid_bound` only constrains the **random UIDs** generated by option 1 -- the `MASTER_KEY` itself can be any value up to `p-1` (approximately 2^252).

**Step 5: Torsion Oracle Range Determination**

Using small UIDs (1, 2, 3, 7, 8, etc.) with the torsion oracle, the mod-8 residues were analyzed. The residue `(M * uid mod p) mod 8` depends on how many times `M * uid` wraps around `p`. Since `p = 5 (mod 8)`, each wrap shifts the residue by 3.

Analysis of the torsion results with various UIDs showed that `M` is in the range `[5p/8, 5p/7)` with `M = 4 (mod 8)` -- approximately 2^250, far beyond the initial BSGS search range.

**Step 6: Binary Search via Continued Fractions**

With `M` narrowed to a range of width approximately 2^249, a binary search was needed to reduce it to a BSGS-feasible range. Each torsion oracle query provides approximately 1 bit of information about `M`.

The `find_best_uid` function finds a UID that creates exactly one "transition" (where `M*uid` crosses a multiple of `p`) near the midpoint of the current interval, effectively bisecting it. It uses:

1. **Continued fraction convergents** of `mid/p` to find UIDs with good balance.
2. **Brute-force near `p/width`** as a fallback (since `uid ≈ p/width` guarantees exactly one transition).

Each query narrows the interval by roughly half, requiring approximately 250 queries total.

**Step 7: Final Enumeration and Flag Recovery**

After 242 binary search steps, the interval was narrowed to fewer than 100 candidates. Each candidate `M = 4 (mod 8)` in the interval was tested by computing `M * G` (where `G` is the Ed25519 generator) and comparing against `Q.y = sign(G_y, 1)` obtained from the server.

Once `MASTER_KEY` was found:
1. Compute `flag_key = M * FLAG_UID % p`
2. Compute `inv_key = flag_key^(-1) mod p`
3. Recover the signed flag point from `flag_sig`
4. Multiply by `inv_key` to get the original flag point
5. Account for torsion ambiguity (8 possible offsets) and x-coordinate sign (2 options)
6. Convert the y-coordinate to bytes and check for `BCCTF{...}` format

### False Starts and Pivots

1. **BSGS in [0, 2^63]**: The initial approach assumed MASTER_KEY was bounded by `uid_bound`. A multi-threaded C implementation of baby-step giant-step searched this range but found nothing -- because M was actually around 2^250.

2. **Invalid curve attack exploration**: Time was spent analyzing whether the `Point(x, y, E)` bug (using mismatched coordinates when padding > 0) could be exploited for an invalid curve attack. While the bug exists, it was not the primary attack vector.

3. **C BSGS implementations**: Several C implementations (`bsgs.c`, `bsgs_mt.c`) were compiled for performance, but the wrong search range made them fruitless.

4. **Torsion oracle consistency confusion**: Early torsion results seemed contradictory until the realization that `p = 5 (mod 8)` causes residue shifts when `M*uid` wraps around `p`, and that M being approximately 5p/8 explained all observations.
